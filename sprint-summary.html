Excalidraw — Sprint Planning Summary

Date: March 8, 2026  ·  Prepared from: 13 open GitHub issues (excalidraw/excalidraw)  ·  Report scope: Issues opened March 1–8, 2026

1. Overview

The current open issue backlog reveals a product under active use but facing reliability concerns in its core drawing experience. The most critical signal is the presence of two independent data loss reports within the same week, suggesting a potential systemic problem with how Excalidraw persists and restores user content. Alongside data loss, performance degradation on Firefox with dark mode and broken drawing input under fast cursor movement indicate that foundational stability work should take precedence over new features this sprint.

2. Key Themes

Theme 1 — Data Loss & Reliability

Two separate users reported losing all their drawing content with no recovery path (#10907, #10915). These are not edge cases — they affect any user who exports, exits, or pauses their session. This is the highest-trust issue in the backlog.

Theme 2 — Core Drawing Interaction Bugs

Three bugs directly affect the act of drawing: cursor speed causing broken strokes (#10876), inverted scaling with shift+scroll (#10898), and lasso selection failing in Compact Mode (#10882). These hit the product's primary value proposition.

Theme 3 — Performance Degradation

Dark theme on Firefox causes severe lag during object manipulation (#10875). With Firefox holding significant browser market share, this is a wide-impact regression that affects everyday use.

Theme 4 — Mobile & End-User UX

The request for a Full Screen button on Android (#10913) reflects growing mobile usage. As Excalidraw is adopted in educational and professional contexts on tablets, mobile experience is becoming a customer expectation, not a nice-to-have.

Theme 5 — File Format & Error Messaging

Issue #10871 exposes a gap in the file loading pipeline where SVG content with a .excalidraw extension produces a cryptic, unhelpful error. The reporter provided a full root cause analysis pointing to blob.ts, making this an actionable and well-scoped fix.

3. Recommended Sprint Priorities

P1 — #10907 & #10915 | Data Loss on Export / Session Exit

Justification: Two data loss reports in one week is a pattern, not a coincidence. Investigate the localStorage persistence layer, export flow, and session restoration logic as a single coordinated effort. Data loss is the highest-severity class of bug — it directly erodes user trust and retention.

P1 — #10875 | Dark Theme Performance on Firefox

Justification: With 4 community comments and a clear reproduction path (enable dark theme → drag/stretch objects), this is a confirmed regression affecting a large user segment. Firefox + dark mode is not an exotic configuration.

P2 — #10876 | Drawing Breaks When Moving Cursor Fast

Justification: The core drawing loop failing under normal fast usage is unacceptable for a drawing tool. This issue also has 4 comments, confirming it affects multiple users across different environments.

P2 — #10871 | Cryptic Error for SVG Files with .excalidraw Extension

Justification: The reporter already diagnosed the exact root cause in blob.ts and proposed a fix. This is a high-value, low-effort issue that can likely be resolved in a single PR and significantly improves the user experience around file loading errors.

P3 — #10913 | Full Screen Button for Android (Chrome)

Justification: Mobile and tablet users are end customers, and maximizing canvas space is a fundamental need for drawing apps on those devices. The Fullscreen API is a well-supported browser standard and implementation complexity is moderate.

4. Risks and Blockers

Data Loss May Be Systemic. The two data loss reports (#10907, #10915) arrived independently, within days of each other, and describe different triggers (export vs. session exit). This raises the risk that the root cause is deeper than a single bug — potentially in the localStorage management, session hydration, or file parsing pipeline. Recommendation: Treat these as one investigation before scoping individual fixes.
No Labels on Any Issues. All 13 issues have zero labels applied. This indicates a triage gap — without labels, there is no automated filtering, no SLA tracking, and no way to query issues by type or component. Recommendation: Establish a labeling convention (e.g., bug, feature, data-loss, performance) and retroactively label these issues.
Spam in the Issue Tracker. Two issues (#10919, #10873) are clearly spam or off-topic. An open issue tracker without moderation creates noise that slows triage. Recommendation: Close both immediately and consider adding an issue template to reduce future spam.
Firefox-Specific Regression (#10875). Performance bugs that are browser-specific are difficult to reproduce in all environments and may require Firefox-targeted profiling. If the team primarily develops on Chrome, this risk may be underestimated.
